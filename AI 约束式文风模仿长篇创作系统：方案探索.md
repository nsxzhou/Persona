# Persona：千面执笔，一念成书。

> **核心定位**：Persona 是一款支持 BYOK（自带 API Key）的单用户约束式创作系统。它包含两个核心模块：
> 
> 1. **风格实验室（Style Lab）**：突破上下文限制，将长篇小说（TXT）清洗、切片并逆向工程，提炼出包含高频词、节奏、视角的结构化风格档案。
> 2. **沉浸工作台（Zen Editor）**：提供极简的无干扰白板。在创作过程中全局挂载风格档案，通过快捷指令呼出受严格文风约束的 AI 辅助续写。在 Persona 中，AI 不再是失控的生成器，而是完全臣服于你审美偏好的文字容器。
>
> **MVP 核心目标**：实现“全局风格档案库”，打通从长篇 txt 样本上传到高质量风格 Prompt 提取的全链路，并完成基础框架（单用户鉴权、API 配置）。

---

## 一、系统重新定义：从“黑盒生成器”到“沉浸式工作台”

基于多次需求收敛，本系统的最终形态被明确为：

1.  **极简的交互哲学**：摒弃复杂的多面板 IDE 布局，采用类似 Notion 的无干扰白板。用户的主要创作动作是打字，并通过 `/` 命令或 Ghost Text（段落末尾触发）呼出 AI 进行风格化续写或改写。
2.  **私有化与极客向**：系统仅支持单用户使用。首次访问时初始化账号密码，用户需自行填入大模型 API Key（BYOK 模式），平台仅作为本地/私有化的调度台，不赚取 Token 差价。
3.  **解耦的“风格实验室”**：文风模仿不与特定小说项目强绑定。系统提供一个全局的“风格实验室”，用户上传几十万字的 txt 样本，系统将其“蒸馏”成一个可复用的**风格档案（Style Profile）**。在任意创作项目中，用户均可挂载该档案作为全局滤镜。

---

## 二、MVP 边界：先做最难的事

在 MVP（最小可行性产品）阶段，我们**暂不**实现复杂的长篇记忆管理（如角色状态自动抽取、剧情时间线维护）和富文本编辑器，而是将火力集中在整个系统最核心、最难的壁垒上：

**MVP 阶段交付物**：
1.  **基础框架**：单用户登录系统、API Key 配置管理、项目管理的基础 CRUD。
2.  **风格实验室**：长文本 txt 上传 -> 切片 -> AI 抽取风格特征 -> 聚合生成结构化 Prompt 的完整工作流。

---

## 三、核心技术挑战：长文本风格如何“蒸馏”？

让 AI 模仿文风，最大的痛点在于：长篇小说（动辄百万字）无法一次性塞进大模型的上下文；即便塞进去（如 Claude 3 200K），模型也很难在海量文本中精准归纳出“文风”，往往会变成“大海捞针”或总结剧情。

因此，**必须构建一条“局部采样 -> 多维提取 -> 全局聚合”的风格提取流水线。**

### 模块 1：样本清洗与分块（Chunking）
- 接收用户上传的 `.txt` 文件。
- 剔除无意义字符、乱码、版权声明等。
- 按章节标记或固定字数（如每 2000 字）将长文本切分为若干个 **文本块（Chunks）**。

### 模块 2：局部特征采样（Sampling）
- 并不是每个文本块都具有强烈的风格特征。系统需采用启发式规则（如对白密度、特定词汇频率）或随机采样策略，从成百上千个 Chunks 中抽取出最具代表性的 N 个片段（如 10 段纯对白、10 段环境描写、10 段战斗/动作场景）。

### 模块 3：多维特征提取（LLM Analysis）
- 调度 LLM 对这些采样片段进行“逆向工程”。
- **提取维度**包括：
  - **词汇习惯**：偏好的高频词、形容词、特定的虚词（如“只觉得”、“猛然”）、罕见或生僻的表达。
  - **句法节奏**：长短句比例、断句习惯、是否有明显的并列结构或排比。
  - **叙事视角与距离**：是贴脸的沉浸式第一人称，还是高高在上的全知全能旁白？
  - **对白模式**：人物说话是简短冷酷，还是长篇大论？是否有特定的语气词？

### 模块 4：特征聚合与 Prompt 生成（Aggregation & Generation）
- 将多轮分析得到的零散特征，再次交给 LLM 进行归纳总结。
- 最终产出一个高度结构化的 **全局风格档案（Style Profile）**。

## 四、核心资产定义：风格档案（Style Profile）

风格提取的最终产物，不是一段简单的提示词，而是一个可以存入数据库、被各个创作项目灵活调用的 JSON 结构。

```json
{
  "profile_id": "uuid",
  "style_name": "金庸武侠风",
  "source_text": "射雕英雄传.txt",
  "global_system_prompt": "你现在是一位武侠小说大师，你的文风特点是：句式简短有力，动词使用极其精准，极少使用华丽的形容词堆砌...",
  "dimensions": {
    "vocabulary": ["高频词1", "高频词2"],
    "rhythm": "长短句结合，动作场面多用四字短句",
    "perspective": "第三人称有限全知"
  },
  "specific_prompts": {
    "dialogue": "在生成对白时，请注意：人物语言应带有文言色彩，少用现代白话...",
    "action": "在描写动作时，侧重于招式的拆解和气机的流动..."
  },
  "few_shot_examples": [
    {
      "type": "environment",
      "text": "风雪满天，破庙中只余一炉残火..."
    },
    {
      "type": "dialogue",
      "text": "郭靖大怒，喝道：‘你这贼子，今日教你有来无回！’"
    }
  ]
}
```

**应用场景**：在未来的 Notion 风格编辑器中，当用户触发“续写”时，系统会将 `global_system_prompt` 以及相关的 `few_shot_examples` 组装成最终的请求发给 LLM。

## 五、基础框架设计（MVP 支撑层）

为了支撑上述重度 AI 调度，MVP 还需要具备以下基础能力：

### 1. 单用户鉴权与初始化
- 首次部署/访问时，进入初始化向导，设置管理员账号和密码。
- 系统不提供注册入口，拒绝任何未经授权的访问。

### 2. BYOK（自带 Key）配置中心
- 提供一个安全的设置面板，用户可填入 OpenAI、Anthropic、DeepSeek 等不同厂商的 API Key。
- 支持在工作流的不同节点配置不同的模型（例如：粗粒度文本切片用便宜的模型，高难度风格聚合和最终正文生成用最强的模型）。

### 3. 异步任务与进度反馈
- 几十万字的文本分析可能需要数分钟甚至更久。
- 后端需引入异步任务队列（如 Celery 或简单的后台线程）。
- 前端需提供可视化的进度条或状态台，告知用户当前处于“正在切片”、“正在提取特征（3/10）”、“正在聚合”等状态。

---

## 六、演进路线（Roadmap）

- **Phase 1 (MVP)**：完成基础框架 + 风格实验室。跑通 txt 到风格档案的提取链路。
- **Phase 2 (Editor)**：实现极简 Notion 风格编辑器。打通 Ghost Text 续写、划词改写，并注入风格档案。
- **Phase 3 (Memory)**：引入“AI 自动抽取代管”。在后台默默维护角色状态、时间线，并在每次生成时自动拼装上下文。

> **核心哲学**：大模型不是小说家，而是一个可训练的风格执行器。我们的目标不是一键生成小说，而是为创作者提供一个能把任何名家文风“据为己有”的超级沉浸式打字机。
