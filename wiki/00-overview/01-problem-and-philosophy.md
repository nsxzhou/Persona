# 01 核心痛点与产品哲学

Persona 的每个设计决策都可以从这一章追溯源头——**五大核心痛点**与**三条产品哲学**。理解它们，才能理解为什么项目的很多地方"有意写得又简单又反直觉"。

---

## 一、五大核心痛点

长篇 AI 创作普遍被以下五个问题卡死。Persona 的架构、模块划分、Prompt 工程策略都在逐一回应它们。

### 痛点 1：语境坍塌与记忆遗忘

**问题**：大模型的上下文窗口有限，百万字小说无法整段塞进 Prompt。早期设定（人物关系、世界规则、伏笔）会随着字数增加被稀释，模型反复"吃书"——每次生成都像失忆，导致设定漂移。

**Persona 的回应**：
- **蓝图 / 活态双层架构（Blueprint + Runtime）**：项目元数据保留在 `projects`，而 5 个蓝图字段与 2 个活态字段统一落在 `project_bibles`；作者手动编辑蓝图层，AI 只提议更新活态层，差异通过 Diff Dialog 人肉确认后入库
- **Prompt Caching + 全量设定注入**：每次生成的 Prompt 包含：系统指令 + 风格母 Prompt + 蓝图层全部字段 + 活态层全部字段（缓存区）+ 轻量局部检索 + 前 3 章原文 + 当前章大纲（动态区）。依赖大模型原生超长注意力 + Prompt Caching，废弃复杂多温区 RAG
- **章节记忆同步（Memory Sync）**：Zen Editor 的 `memory-sync-button.tsx` 允许用户在写完一章后批量触发活态层回灌，把新揭示的设定自动累加

详见 [23 圣经与世界观](../20-domains/23-bible-worldbuilding.md)、[30 记忆同步](../20-domains/30-memory-sync.md)。

### 痛点 2：叙事逻辑脱节

**问题**：大模型本质是自回归文字接龙机器，没有远期规划与因果链追踪能力。挖了坑不填、人物动机前后矛盾、伏笔开了就忘。

**Persona 的回应**：
- **总纲 → 分卷 → 分章 → 节拍（Beat）四级结构**：在 Outline 系统里，作者先写出总纲与分章，然后在 Zen Editor 写作前对当前章生成 5–10 条节拍点（每条一句话），确认后才逐段生成正文
- **伏笔追踪（`runtime_threads`）**：蓝图字段里单独给"活跃伏笔 / 已回收线索 / 设定约束备忘"留了位置，每次续写时模型读到，提议更新时作者审查
- **节拍驱动的结对写作（Beat-Driven Co-Creation）**：把"大段黑盒生成"拆成"先生成本章节拍 → 人类确认 → 分段流式生成（每段约 500 字）"的增量循环

详见 [24 大纲与节拍](../20-domains/24-outline-and-beats.md)。

### 痛点 3：AI 味与叙事扁平化

**问题**：经过 RLHF 对齐的模型倾向于"快速平淡地解决冲突"，且遣词造句极易向"AI 均值"滑落（滥用"的"、"了"、高频套话、机械句式）。写出来的文字"读起来像 AI 写的"。

**Persona 的回应**——两条并行策略：

1. **风格前置（Shift-Left Styling）**：消除 AI 味的最佳时机是**起草阶段**而不是事后清洗。Style Lab 对长篇样本做深度风格分析，产出 `prompt_pack_markdown`——这个 Markdown 文件作为 **System Prompt 最高优先级**注入所有续写请求。大模型凭借上下文模仿能力（"Show, don't tell"）直接按目标语感/句式节奏/标点/方言/口头禅生成初稿
2. **局部重写（Inline Copilot）**：大模型生成的文本直接上屏，不经过任何自动 Linter 拦截。作者是唯一质量评审员，遇到 OOC 或文笔平淡的段落使用 `Cmd+K`（划词改写）做局部 Reroll 或提供具体修改指令

关键是：**废弃事后清洗**。很多同类产品花大量算力做"自动消痕"、"五维流水线评审"、"AI 味检测器"——Persona 极客式地把这些删了，把算力还给作者的实时交互。

详见 [26 Style Lab](../20-domains/26-style-lab.md)、[30 Prompt 语料总览](../30-prompt-engineering/30-prompt-overview.md)。

### 痛点 4：战力与体系崩坏（网文/奇幻领域特化）

**问题**：没有数值约束的模型很容易破坏奇幻/网文里严密的等级晋升体系——主角突然跨级、设定上写好的战力天花板被随手破掉。

**Persona 的回应**：
- 目前主要通过蓝图字段 `world_building` + `characters` 把战力/等级体系作为硬约束写进 Prompt
- 长期规划中 **Style Generator Graph** 会加入 **Critic Node**，对生成结果做硬规则校验（战力、忌讳词、句长分布、风格偏差）

目前仍属于"前置约束 + 人肉审校"兜底，没有自动化数值校验器。这是有意的——作者对自家设定的边界感远好于自动规则。

### 痛点 5：创作过程不可控（黑盒生成）

**问题**：一键生成一大段文本的方式**试错成本极高**，作者无法像写代码一样微调与回滚。生成 2000 字后发现第一段节奏错了——整段丢掉重生成？还是凭记忆找问题局部修？

**Persona 的回应**：
- **增量生成**：节拍驱动的写作把一章拆成 5–10 个节拍、再把每个节拍拆成 ~500 字段，作者可随时暂停、接手、回滚
- **Diff 确认**：蓝图活态层的 AI 提议更新不会直接落库，而是弹 Diff Dialog 让作者逐字段审查后保存
- **明确的 draft/confirmed 分层**：Zen Editor 里的章节正文与圣经/大纲字段都有"草案 vs 确认"的边界，作者说了算

## 二、三条产品哲学

从痛点推导出的三条产品层面的硬约束。

### 哲学 1：极简交互

**摒弃多面板 IDE 式布局**，优先保证创作过程的沉浸感。

这一点体现在：

- Zen Editor 命名就来自 "Zen Mode"——核心写作页面没有工具栏爆炸、没有三列布局、没有悬浮窗轰炸
- 圣经、大纲、设置放在 **Tab 折叠面板**里，默认隐藏
- 前端**不使用**富文本编辑器（如 TipTap、ProseMirror、Lexical），而是**原生 textarea**——理由：富文本编辑器的"AI Ghost Text 续写 + 局部改写 + 节拍对齐"组合交互实现代价极高，MVP 不值得
- 前端组件库用 **shadcn/ui**（基于 Radix + Tailwind），保持默认审美，不做花哨主题

### 哲学 2：私有化与极客向

- **不提供公开注册入口**：首次访问会被路由守卫重定向到 `/setup`，管理员一次性初始化
- **单用户模型**：只有一个 `user` 记录，所有业务资源通过 `user_id` 做 scope，但实质上只服务一个人
- **BYOK（Bring Your Own Key）**：不代理 LLM 调用、不包装计费，作者自己填 OpenAI / Anthropic / 本地 LLM 的 endpoint + key，花的是作者自己的额度
- **本地优先**：Postgres + 本地文件系统（`PERSONA_STORAGE_DIR`）。风格样本 TXT、分析日志、导出产物都落本机

### 哲学 3：全局风格资产

**Style Profile 不强绑定某个项目，而是全局资产**。

- 数据库表 `style_profiles` 按 `user_id` scope，不含 `project_id` 外键
- 同一个作者可以有多个风格档案（"金庸武侠风"、"余华冷峻风"、"村上春树风"）
- 项目通过 `projects.style_profile_id` 挂载一个档案，重新挂载只是外键替换——不会拷贝 Voice Profile
- 这意味着：**风格档案一旦编辑，所有挂载它的项目立刻生效**——作者可以集中维护一份风格，多个项目共享

这个设计背后的假设：**作者的风格是跨作品的，不是每本书都要独立 Style Lab 一次**。

## 三、明确反对的做法

列出 Persona **故意不做**的事情——这些在同类产品里很常见，但 Persona 认为它们要么过度工程化、要么方向错了。

### ❌ 多 Agent 流水线 + 五维流水线评审

很多类似产品搞出"写手 Agent + 审校 Agent + 编辑 Agent + 读者 Agent + 主编 Agent"的多 Agent 流水线，做自动五维打分。Persona 认为：

- **成本爆炸**：一段 500 字生成跑 5 轮 LLM，每章 ~100 段就是 500 次 LLM 调用
- **效果不可靠**：自动打分容易把"读起来更像 AI 的稳妥产出"判为"更好"
- **替代方案**：前置约束（风格档案）+ 作者审校（Inline Copilot）足够

### ❌ 自动消痕（De-AI-ification）后处理

自动检测并替换"AI 高频词"、"AI 套话"、"机械句式"。Persona 认为：

- **替换质量无保障**：自动替换器本身是模型，会引入新的 AI 味
- **破坏作者语感**：某些"AI 高频词"恰好是某个作者的常用词
- **替代方案**：Style Lab 的 Markdown-First Voice Profile 在起草阶段就约束词汇——**从源头省掉"后处理"步骤**

### ❌ 复杂多温区 RAG + 知识图谱自动维护

很多系统搞出"热记忆 / 暖记忆 / 冷记忆"多温区存储 + 自动从文本抽实体建图谱。Persona 的选择：

- **手动编辑的 KG**：蓝图字段是纯文本（不是结构化 KG），作者自由维护
- **Prompt Caching 取代 RAG**：依赖大模型原生长上下文 + Prompt Caching 缓存区技术，全量设定直接注入
- **轻量局部检索**：仅在必要时提取当前章大纲的关键词，从前文捞一小段——作为补充，不强制

这条路在 Gemini 2.0 / Claude 3.5 / GPT-4.1 这一代大模型（1M+ context、Prompt Caching 原生支持）上非常划算。

### ❌ 多用户 SaaS / 公开注册 / RBAC / 计费

见哲学 2。Persona 只是私有化的极客工具。

### ❌ 富文本编辑器 + WYSIWYG

见哲学 1。Markdown / 纯文本 + 外部预览器。

### ❌ 大模型自由发挥

从 `ANALYZE-GENERATE.md`（详见 [32](../30-prompt-engineering/32-analyze-generate-playbook.md)）：

> 你是一位**文风复制引擎**。你的任务**不是自由发挥**，而是在给定写作任务内，最大化复现目标文本的风格约束、自然度和局部可控性。

Persona 把大模型显式当作**受约束的执行器**，不期待模型"创作"，只期待模型"按约束生成"。

---

## 四、一句话总结哲学

> 大模型不是小说家，而是一个可训练、可约束、可审校的**风格执行器**。

这是 Persona 所有架构决策的落脚点。每一篇后续文档都可以用这句话回溯。
